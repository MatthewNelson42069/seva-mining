### Semantic-token consumption (D-07 + D-08 strict requirement)

**All new code introduced by Phase 9 MUST consume the semantic tokens, NOT literal `amber-*` classes:**

| New code surface | Required token | Forbidden literal |
|------------------|----------------|-------------------|
| Active switcher button border | `border-brand-accent` | ~~`border-amber-500`~~ |
| Active switcher button text | `text-brand-accent` | ~~`text-amber-500`~~ |
| Active switcher button background tint | `bg-brand-accent-subtle` | ~~`bg-amber-500/5`~~ |

**Rationale:** Phase 8 D-05 established the semantic-token namespace for new code to consume so v3.1+ per-tenant branding can swap the `--brand-accent` value per tenant via a single CSS variable update (e.g., Juno could later get `oklch(0.6 0.2 240)` blue) WITHOUT touching every component. Phase 9 is the FIRST consumer of this contract — getting it right here sets the v3.x precedent.

**Existing literal `amber-*` usages in the freeze-lifted AppHeader (brand-mark square `bg-amber-500`) and frozen TabNav (`border-amber-500 text-white` active state) remain unchanged** — Phase 8 D-05 explicit guidance is "do not sweepingly refactor existing usages." ### CompanySwitcher canonical contract (D-07 + D-08 — LOCKED)

**Visual layout (segmented control, NOT dropdown, NOT Cmd+K palette):**

```
+----------------------------------------------------------+
|  [S] Seva Mining       [Seva][Juno]            Log out   |
+----------------------------------------------------------+
   |____ frozen ____|    |__ new __|    |_____ frozen ____|
```

**Default (inactive) button state:**
- Classes: `px-3 py-1.5 text-sm font-medium rounded-md border border-zinc-800 text-zinc-400 transition-colors`
- Background: transparent (inherits AppHeader `bg-zinc-900`)
- No `bg-*` utility on the button itself

**Hover (inactive) state:**
- Adds: `hover:border-zinc-700 hover:text-zinc-100`
- Transition: `transition-colors` (Tailwind default 150ms)
- No layout shift — border thickness identical to default state

**Active state (derived from `useParams().company === thisButton.company`):**
- Classes: `px-3 py-1.5 text-sm font-medium rounded-md border border-brand-accent text-brand-accent bg-brand-accent-subtle transition-colors`
- Active state has NO hover variant — already at peak emphasis; further hover lift would compound visual weight and confuse mouse-down feedback

**Focus state (keyboard navigation):**
- Adds: `focus:outline-none focus-visible:ring-2 focus-visible:ring-brand-accent focus-visible:ring-offset-1 focus-visible:ring-offset-zinc-900`
- WCAG 2.1 AA compliant focus indicator (≥3:1 contrast against surrounding zinc-900 — accent ring is `oklch(0.769 ...)`, ratio ~8.5:1)

**Element type:** `<button type="button">` — NOT `<a>` / `<NavLink>`. Reason: clicking must (a) call `queryClient.clear()` before navigation (D-08) AND (b) call `navigate(`/${nextCompany}${currentSubPath}`)` with computed sub-path. A `<NavLink>` would do the navigation but not the cache clear; using `<button>` keeps both side effects atomic in one onClick handler.

**Tab preservation on switch (D-07 + Claude's Discretion clarification):**
- `useLocation().pathname` is parsed: strip leading `/${currentCompany}`; remainder is `currentSubPath` (e.g. `/calendar`, `/viral`, or `''` for index)
- `navigate(`/${nextCompany}${currentSubPath}`)` — e.g. `/seva/calendar` → `/juno/calendar`
- Edge case: `currentSubPath === ''` (operator is on `/seva/`) → navigates to `/juno/` (Juno's index tab, which renders the Juno News Funnel empty-state in v3.0)
- Edge case: Juno doesn't have content for the current sub-path (e.g. user switches from `/seva/viral` to `/juno/viral`) → lands on `/juno/viral` which renders the v3.1 empty-state. This is intentional per CONTEXT D-claude — operator's mental model is "I'm on the calendar/viral tab; just different company now."

**Tooltip on hover:** None in v3.0. Two visible labels make a tooltip redundant. v3.1+ may add tooltips on icon-only variant if introduced.

**Keyboard shortcut:** Not in v3.0. CONTEXT deferred Cmd+1 / Cmd+2 to v3.1+ (FEATURES.md "Should have" tier). Phase 9 ships click-only.

**Three tabs render three slightly-different copies. All use the same visual treatment.**

### Keyboard navigation contract

- **Tab order** in AppHeader: `[brand-mark group is non-focusable]` → `[Seva button]` → `[Juno button]` → `[Logout button]`. Brand-mark is a `<div>` (not focusable) — matches frozen Phase 5 baseline.
- **Enter or Space** on focused switcher button: activates the button (same as click).
- **Tab from "Juno" button**: moves focus to "Log out" button (then onward to page content via natural tab order).
- **Shift+Tab from "Seva" button**: moves focus backward to the previous focusable element (typically the URL bar or nothing, depending on starting point).
- **No roving tab index pattern** — both switcher buttons are individually tab-focusable. This is correct for a 2-element segmented control (matches the WAI-ARIA "Group of Buttons" pattern, not "Tab Bar" which would imply roving). The buttons are `<button>` elements, not `role="tab"`.
- **No ARIA labels required** — visible text labels "Seva" / "Juno" are accessible names. Optionally add `aria-current="page"` to the active button for screen readers; planner confirms in 09-PLAN.md.
