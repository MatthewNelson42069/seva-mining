# Quick Task 260424-e37: Replace sub_gold_history's curated 14-JSON pool with online SerpAPI fetch for fresh historical gold stories

## User's verbatim ask

> "Gold history agent should use historical stories. But it should do an online fetch to make sure it gets new/fresh one"

Clarifying answers given in-turn:

> Q1 (fetch source): "Could be either historical story topics or achival sources"
> Q2 (fact source): "Article text is good enough for fact source"
> Q3 (pool strategy): "Whatever you think. I just want an interesting historical gold story every other day"
> Q4 (format): "Stays thread + carousel"

Pool-strategy discretion ("Whatever you think") transfers decision authority for the 10 gray areas below to this CONTEXT doc.

## One-line intent

Replace sub_gold_history's curated 12-JSON fact-sheet pool with an online SerpAPI fetch that surfaces fresh historical gold-sector stories each run, using the article's own text as the sole fact source (mirrors breaking_news / infographics pattern). Cadence, format, drafter voice, item-per-run cap all unchanged.

## Design shift (core idea)

**Before (curated pool — lbb pattern):**
```
_pick_fresh_slug(whitelist_of_12_slugs, used_slugs_from_Config)
    → load_fact_sheet(slug) from JSON
    → _draft_gold_history(fact_sheet)       # drama-first narrative
    → gold-gate (Haiku)
    → review (Haiku)
    → persist
```

**After (online fetch — lvp / breaking_news pattern):**
```
_select_historical_gold_queries(count=3, buffer=0)         # day-seeded shuffle of ~12 queries
    → fetch_analytical_historical_stories(queries)          # reuse lvp helper; SerpAPI
    → dedup against UsedURL (new Config row of canonicalized URLs)
    → loop candidates (best-scored first):
        → gold-gate (Haiku, existing)
        → _draft_gold_history(article_text + headline + url)  # drafter input SHAPE shifts
        → review (Haiku, existing)
        → persist (ContentBundle + DraftItem)
        → record URL into UsedURL dedup set
        → break on first success (cap=1/run)
```

## Locked decisions (10 gray areas)

### Q1 — Pool handling: replace entirely vs keep as fallback?
**LOCKED: (C) — Delete the 12 JSONs and the `gold_history_stories/` directory entirely. Repurpose the existing `Config.gold_history_used_topics` row for URL-based dedup (rename the key to `gold_history_used_urls`).**

Rationale: user said "fresh/new", so the curated content is no longer the product. Preserving the dedup infrastructure (Config-row list) gives us all-time no-repeat on URLs without a schema change. 12 obsolete JSONs + `load_all_stories` + `load_fact_sheet` + `_pick_fresh_slug` all become dead code and are removed. The existing slug rows in prod's Config become irrelevant (new code writes URLs to a new key); migration strategy: simply introduce the new key, leave the old one as dormant archival data. No Alembic needed.

### Q2 — Dedup horizon
**LOCKED: (A) — All-time URL dedup.**

Rationale: gold_history runs every other day (183 runs/year). Retelling the same Klondike article six months later would be transparently repetitive. URL is the canonicalized dedup key (lowercase + stripped tracking params). Stored as JSON list in `Config.gold_history_used_urls`. Unbounded growth is acceptable (≤200 entries/year).

### Q3 — Drafter prompt surgery (input-shape change)
**LOCKED: (B) — Surgical prompt edits.**

Rationale: the current `_draft_gold_history` prompt references the `verified_facts` list + `sources` array from the JSON fact-sheet shape. Those references become invalid after the input shape changes to `{article_text, headline, url, source_name}`. Leaving stale references would confuse Sonnet.

Required surgery:
- Swap references to `verified_facts` → `article_text` throughout system + user prompt
- Drop references to `sources` (list of URLs) → single `url` + `source_name`
- Strengthen the no-hallucination instruction: "Base every specific claim on statements literally present in `article_text`. Do NOT fill in details from general historical knowledge. If `article_text` is thin, write a thinner thread rather than embellishing."
- Preserve: drama-first narrative voice, arc guidance, brand voice, carousel-slide structure.

### Q4 — Gold gate filtering
**LOCKED: (A) — Reuse existing `is_gold_relevant_or_systemic_shock` Haiku gate.**

Rationale: the existing gate is the right tool. Historical narratives of gold's ups-and-downs should pass because they concern gold directly. The bullish-only j7x filter runs alongside; it already distinguishes historical factual declines (allowed) from forward-looking bearish forecasts (blocked). If in production this gate rejects too aggressively, we'll tune it in a follow-up — out of scope here.

### Q5 — Quantity cap per run
**LOCKED: (A) — Keep `items_queued = 1` per run.**

User's exact words: "an interesting historical gold story every other day". 1 per run × every-other-day cron = 0.5/day average.

### Q6 — Candidate fetch depth (# of SerpAPI queries per cycle)
**LOCKED: (A) — 3 queries per cycle via day-seeded deterministic shuffle.**

Rationale: `fetch_analytical_historical_stories` sends 5 results per query → 3 queries = ~15 candidate articles. After dedup + gold-gate + review, this gives enough depth to find at least one passer while keeping SerpAPI spend at ~45 calls/month (negligible vs. breaking_news's 2-hour cadence).

Implementation: `_select_historical_gold_queries(count=3, seed=date.today().toordinal())` — mirrors lvp's `_select_analytical_queries`.

## Dedup strategy details

**New Config key:** `gold_history_used_urls` (typed as JSON list of canonicalized URL strings).
- Canonicalization: `url.lower().split('?')[0].rstrip('/')` — strips query strings (trackers) and trailing slashes; case-insensitive.
- Read path: helper `_get_used_urls(session) -> set[str]`.
- Write path: helper `_record_used_url(session, url)` — append if not already present.
- Both mirror the existing `_get_used_topics` / `_record_used_topic` shapes in the current file; just change the key name and the canonicalization.

**Old key:** `gold_history_used_topics` — leave dormant. Prod has some rows from lbb; they're now archival data. New code doesn't read them. No Alembic.
