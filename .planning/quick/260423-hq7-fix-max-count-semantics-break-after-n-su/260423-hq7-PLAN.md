<objective>
Fix the `max_count` semantics bug in `scheduler/agents/content/__init__.py::run_text_story_cycle` so it means "break after N successful drafts" instead of "trim candidates to top N before the dedup loop."

Currently the trim at ~L177-193 happens BEFORE `_is_already_covered_today` runs. When the top-N-by-recency candidates have already been drafted by another sub-agent (e.g. breaking_news ran first at 9:25 AM and drafted Kitco/FXStreet/etc.), threads/long_form/quotes enter the loop at 9:51 AM / 10:08 AM / 12:00 PM with N already-covered candidates, the loop `continue`s N times, items_queued stays 0, and the cycle produces zero drafts even when candidates 3-20 are fresh and gold-relevant.

Empirical evidence the bug is live right now:
- sub_breaking_news 9:25 AM PDT: 6 queued (fresh pool)
- sub_breaking_news 9:34 AM PDT: 0 queued (pool drained by BN)
- sub_threads 9:51 AM PDT: 0 queued
- sub_long_form 10:08 AM PDT: 0 queued
- sub_quotes 12:00 PT: 0 queued
- sub_breaking_news 11:34 AM PDT: 2 queued (new stories arrived — proves it's not slow news)
- Threads/long_form: 20+ consecutive "No content found" runs going back to 2026-04-22

Purpose: Unbreak 4 of the 5 text sub-agents (threads, long_form, quotes, infographics) so they produce drafts on every cycle with fresh stories available, without regressing breaking_news (max_count=None) or the of3/d30 invariants (sort_by="score", source_whitelist, dedup_scope="same_type").

Output: Single atomic commit on `main` changing only `scheduler/agents/content/__init__.py` and `scheduler/tests/test_content_init.py`. No caller module touches. No schema changes. No frontend touches.
</objective>

<decision_candidate_cap>
The task description flags a possible MAX_DRAFT_ATTEMPTS safety cap (e.g. 15) to prevent runaway Claude spend in a pathological case where dedup/draft rejects nearly everything.

**Decision: SKIP the candidate cap for this fix.** Rationale:
1. Existing upstream gates (gold_gate Haiku classifier, source_whitelist for quotes, _is_already_covered_today 85% fuzzy) already bound the candidate set in practice — fetch_stories() returns roughly 20-40 items per cycle after the 30-min cache, not thousands.
2. Adding an unrelated safety cap in the same commit expands blast radius and obscures the intent of this fix ("remove pre-loop trim, add post-persist break"). If runaway spend materializes, it gets its own focused quick task.
3. The real attack-surface for spend is `content_agent.fetch_stories()` volume, not the dedup loop — a cap here would paper over the wrong layer.

Surface-level telemetry in the existing `agent_run.notes` JSON ("candidates": len(candidates)) already gives visibility into how many candidates the loop evaluated per run if post-deploy observation flags a spend concern.
</decision_candidate_cap>
