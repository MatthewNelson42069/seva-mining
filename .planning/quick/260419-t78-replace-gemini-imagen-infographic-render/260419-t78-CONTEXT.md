# Quick Task 260419-t78: Replace Gemini Imagen infographic rendering with Claude chart_spec + Recharts - Context

**Gathered:** 2026-04-20
**Status:** Ready for research + planning

<domain>
## Task Boundary

Replace Gemini Imagen infographic rendering with Claude chart_spec + Recharts (a16z-style).

The content agent currently emits a free-form "draw an infographic" prompt for Gemini Imagen.
Output is inconsistent (often illustrative instead of data-dense, text frequently garbled).
Gemini produces 4 images per bundle (1 Twitter 16:9 + 3 Instagram slides) — only the Twitter
image is consumed; Instagram was purged in quick-lvy.

This task replaces the **infographic branch** of `image_render_agent.py` with:
1. Content agent emits structured `chart_spec` JSON (validated via Pydantic) when format=infographic
2. Recharts SSR renders the spec into SVG in a Node-based chart renderer
3. resvg-js rasterizes SVG → PNG; scheduler uploads PNG to R2
4. Existing `rendered_images: [{role: 'twitter_visual', url, ...}]` shape preserved — no frontend changes required
5. Drop the 3 `instagram_slide_*` roles (dormant post-lvy)

**Out of scope:**
- Quote format stays on Gemini for this iteration
- Breaking news / thread / long_form formats (no image needed)
- Frontend display changes (ContentSummaryCard gallery already wired in quick-si2)

</domain>

<decisions>
## Implementation Decisions

### Design system + visual style (locked pre-discussion)
- **Use Claude design tokens** (researcher to validate what's published — user noted recent update)
- **Base visual style on a16z published infographics** (researcher to capture color palette, typography hierarchy, chart preferences, headline placement)

### Chart library (locked pre-discussion)
- **Recharts** — the declarative React charting library. Emits SVG natively (required for resvg-js pipeline).

### Carousel rule (locked pre-discussion → refined in discuss 4)
- User: "If it needs two charts to get the points across, do it"
- **Agent decides via prompt** — Sonnet emits `charts: [spec1, spec2]` when story genuinely needs two views. Prompt includes explicit guidance on when to split (e.g. time dimension + category dimension can justify two charts).
- No hard data-shape heuristic, no type-pair auto rules. Trust the model with guidance.

### Quote format (locked pre-discussion)
- **Stays on Gemini Imagen** — this task does NOT migrate quotes. Explicit scope exclusion.

### Gray area 1 — PNG rendering location
- **Decision: Server-side resvg-js (Node)**
- Add a tiny Node-based chart renderer sidecar to the scheduler worker (or a dedicated Railway service if worker can't easily carry Node)
- Pipeline: `chart_spec` → Recharts SSR → SVG string → `@resvg/resvg-js` → PNG buffer → R2 upload
- **Rejected alternatives:**
  - Playwright (~200MB image bump, slow cold start — overkill)
  - Client-side html2canvas (image doesn't exist until operator approves; font/raster fidelity risk across browsers)
  - Python cairosvg (no Recharts = we'd have to re-implement the chart types by hand in Python — too much work)

### Gray area 2 — Fallback when Sonnet can't produce valid chart_spec
- **Decision: Downgrade to thread (text only)**
- If Pydantic validation fails on Sonnet output, log failure and set `format: thread` for the bundle
- Bundle ships with no image — operator sees a clean text post, not a broken chart
- Never retry with Gemini (reintroduces the inconsistency we're killing)
- Never hard-fail the bundle (too aggressive for a weekly occurrence)

### Gray area 3 — chart_spec schema enforcement
- **Decision: Pydantic v2 model shared between scheduler + backend**
- Define `ChartSpec`, `ChartDataPoint`, etc. as Pydantic models in shared location (e.g. `scheduler/models/chart_spec.py` imported by backend)
- Scheduler validates Sonnet output via `ChartSpec.model_validate_json(...)`
- Backend API response serializes the spec via the same model (so frontend can access it if ever needed)
- Single source of truth, type-safe, additive evolution via field optionality

### Gray area 4 — Carousel decision logic (refined)
- **Decision: Agent decides via prompt**
- Sonnet's content-drafting prompt includes: "Emit `charts: [spec1, spec2]` when the story genuinely needs two views to convey the point (e.g. time-trend + category-breakdown). Default to a single chart."
- No hard-coded rules — trust the model with explicit guidance
- Rendering pipeline handles `charts: [...]` uniformly (emits 1 or 2 PNGs accordingly)

### Claude's Discretion
- Exact Pydantic schema field names and nested structure (researcher will confirm Recharts component signatures match a natural JSON representation)
- Specific Claude design tokens to adopt (researcher will identify what's published; if no formal system, use a16z aesthetic as primary source)
- Node sidecar architecture (in-worker subprocess vs separate Railway service) — depends on research finding on Railway Docker layer sizing
- Exact prompt wording for Sonnet's chart_spec emission (planner will draft, grounded in research)
- Chart type coverage in v1 — all 8 listed are aspirational; researcher may flag stat_callouts or comparison_table as needing custom components (Recharts is chart-focused, not layout-focused)
